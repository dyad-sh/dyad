# Supabase Database Rollback via Undo-SQL

> Generated by swarm planning session on 2026-03-18

## Summary

Enable database schema rollback for Supabase users by automatically generating reverse (undo) SQL for each DDL change executed through Dyad. When a user reverts to a previous version, the accumulated undo-SQL is executed in reverse chronological order, rolling back the database schema alongside the code. This closes the feature gap where Neon users get automatic database rollback via time-travel but Supabase users only get code reverted.

## Problem Statement

Today, Supabase users in Dyad have no way to roll back their database when reverting to a previous version. The version revert flow — a core Dyad differentiator ("undo for your whole app") — is fundamentally broken for Supabase: code reverts to version N, but the database stays at version N+5. The app may not even start. Neon users get seamless time-travel restore, but Supabase users get nothing. This creates an inconsistent and frustrating experience for a significant segment of Dyad's user base.

## Scope

### In Scope (v1 — Undo-SQL for DDL)

- Automatic undo-SQL generation for DDL (schema changes) at SQL execution time
- Storage of undo-SQL per version in the `versions` table
- Multi-version rollback: compose undo-SQLs in reverse chronological order
- Transaction-wrapped execution (all-or-nothing)
- Confirmation modal with human-readable summary and expandable raw SQL
- Multi-step progress stepper during rollback (Code → Database → Edge Functions)
- DB rollback indicator on version history rows
- Public schema only — warn on non-public schema changes
- Graceful degradation: if undo-SQL generation fails, version is marked code-only

### Out of Scope (v2+)

- Full database dumps (per-version or pre-rollback)
- DML (data change) rollback (INSERT/UPDATE/DELETE undo)
- Database preview mode for Supabase (no branch isolation like Neon)
- Cross-provider rollback abstraction
- Supabase branching integration (paid tier only)
- Storage/retention management UI
- Execution result storage for future DML undo

## User Stories

1. **As a Dyad user with Supabase**, when I revert to a previous version, my database schema rolls back along with my code, so the app works at the reverted version.
2. **As a Dyad user**, I see a clear warning before rollback that data in dropped tables or columns will be permanently lost.
3. **As a Dyad user**, I can see in the version history which versions have database rollback available vs. code-only.
4. **As a Dyad user**, I never notice undo-SQL generation happening — it adds no latency or friction to my normal app building workflow.
5. **As a Dyad user**, if database rollback fails during a revert, the transaction rolls back, my database is unchanged, and I see a clear error with the option to revert code-only.
6. **As a Dyad user**, versions created before this feature was added show "No DB rollback available" explicitly, not an ambiguous absence of indicator.

## UX Design

### User Flow

1. User has a Supabase project linked to their app (existing flow)
2. AI makes schema changes via `executeSupabaseSql` → undo-SQL is silently generated and stored per version (no UI, no setting)
3. User opens Version History pane → versions with undo-SQL show a DB badge (reusing existing Neon badge pattern)
4. User clicks "Restore" on a previous version
5. Confirmation modal appears with:
   - Code revert summary (existing)
   - Database changes summary: "The following schema changes will be undone:" with human-readable descriptions
   - Collapsible raw SQL preview (using existing `CodeHighlight` component)
   - Warning: "Data in dropped tables or columns will be permanently lost."
   - Note if RLS/policy changes are included: "Includes security policy changes"
6. User confirms → progress stepper shows: Reverting code... → Applying database changes... → Redeploying edge functions...
7. Success toast: "Restored to Version X. Database schema reverted (N statements applied)."

### Key States

- **Default (version with undo-SQL)**: DB badge visible on version row. Tooltip: "Database schema can be rolled back for this version"
- **No undo available**: Grey/absent badge. Tooltip: "No database rollback — created before rollback support" or "Contains SQL that can't be automatically undone"
- **Chain broken**: If any intermediate version lacks undo-SQL, show clear message: "Cannot revert database — version X contains changes that can't be automatically undone. Code-only revert is available."
- **Loading**: Progress stepper with per-phase status, version pane stays open during restore
- **Success**: Toast with specific counts: "Database schema reverted (N statements applied)"
- **Partial failure**: "Code restored to Version X. Database rollback failed: [error]. Your database schema may not match your code." Chat messages preserved on partial failure.
- **Error**: Transaction rolled back, database unchanged, clear error message with option to revert code-only

### Interaction Details

- Confirmation modal uses destructive button styling (red/warning color) for "Restore"
- SQL preview uses same `CodeHighlight` component from chat messages for consistency
- Skippable via existing consent pattern (ask/always/never) for power users
- Version pane stays open during restore (override current `onClose()` behavior)
- Message deletion gated on full success — preserve chat context on partial failure

### Accessibility

- Confirmation modal traps focus and is keyboard-dismissible (Escape)
- Restore button has clear `aria-label` distinguishing code-only vs. code+DB restore
- DB badge does not rely on color alone — includes text/icon
- Progress status announced via `aria-live="polite"` region
- SQL preview in `<pre>` block with proper language semantics
- All new interactive elements have visible focus indicators

## Technical Design

### Architecture

Hook into the existing SQL execution interception in `response_processor.ts` (lines 178-211). When SQL executes successfully against a connected Supabase project, generate the inverse DDL via a new pure-function undo-SQL generator module. Store the undo-SQL on the version record. On revert, collect undo-SQLs for all intervening versions, compose in reverse chronological order, and execute in a single transaction via `executeSupabaseSql`.

### Data Model Changes

Single new column on the existing `versions` table:

```sql
ALTER TABLE versions ADD COLUMN supabase_undo_sql TEXT;
```

In Drizzle (`src/db/schema.ts`):

```ts
supabaseUndoSql: text("supabase_undo_sql"),
```

The presence of a non-null `supabase_undo_sql` value indicates DB rollback is available for that version. No additional columns needed for v1.

### API Changes

No new IPC contracts. Existing contracts extended:

- `listVersions` response: add `hasDbUndo: boolean` field per version
- `revertVersion`: internally extended to handle Supabase undo-SQL execution. Returns existing `successMessage | warningMessage` format.
- Progress events emitted via existing `safeSend` IPC event pattern during multi-step revert

### Undo-SQL Generator — Supported DDL Patterns

| Forward SQL                                 | Undo SQL                                                |
| ------------------------------------------- | ------------------------------------------------------- |
| `CREATE TABLE <name> (...)`                 | `DROP TABLE IF EXISTS <name>`                           |
| `ALTER TABLE <t> ADD COLUMN <col> <type>`   | `ALTER TABLE <t> DROP COLUMN <col>`                     |
| `CREATE INDEX <name> ON <table>`            | `DROP INDEX IF EXISTS <name>`                           |
| `CREATE POLICY <name> ON <table>`           | `DROP POLICY IF EXISTS <name> ON <table>`               |
| `ALTER TABLE <t> ENABLE ROW LEVEL SECURITY` | `ALTER TABLE <t> DISABLE ROW LEVEL SECURITY`            |
| `CREATE OR REPLACE FUNCTION <name>`         | `DROP FUNCTION IF EXISTS <name>`                        |
| `CREATE TRIGGER <name>`                     | `DROP TRIGGER IF EXISTS <name> ON <table>`              |
| `DROP TABLE`                                | Not reversible → `null` (warn/skip)                     |
| `ALTER TABLE DROP COLUMN`                   | Not reversible without stored data → `null` (warn/skip) |
| Unrecognized SQL                            | `null` (warn/skip)                                      |

**Conservative principle**: When in doubt, return `null` rather than wrong undo-SQL. A missing undo is a UX inconvenience; a wrong undo is data corruption.

## Risks & Mitigations

| Risk                                                         | Likelihood | Impact | Mitigation                                                                                                         |
| ------------------------------------------------------------ | ---------- | ------ | ------------------------------------------------------------------------------------------------------------------ |
| Undo-SQL incorrect for complex DDL                           | HIGH       | HIGH   | Conservative generator (return null on uncertainty), transaction-wrapped execution, comprehensive unit tests       |
| Multi-version undo chain broken (intermediate null)          | MEDIUM     | MEDIUM | Pre-check all intermediate versions before starting, fail fast with clear message                                  |
| Supabase `runQuery` API payload/timeout limits on large undo | MEDIUM     | MEDIUM | Test with realistic multi-version rollbacks; fall back to per-version `runQuery` calls if needed (loses atomicity) |
| Partial rollback leaves app in inconsistent state            | LOW        | HIGH   | Transaction wrapping ensures all-or-nothing; message deletion gated on full success                                |
| RLS policies/triggers create ordering dependencies           | MEDIUM     | MEDIUM | Generator handles policy/trigger drop before table drop; flag in confirmation modal                                |
| Non-public schema changes (auth/storage) not covered         | LOW        | LOW    | Explicitly exclude non-public schemas; log warning for user visibility                                             |
| Retroactive coverage gap (existing versions)                 | CERTAIN    | LOW    | Clear UI indicator: "No DB rollback — created before rollback support"                                             |

## Open Questions

- **Supabase `runQuery` transaction support**: Does the Management API support `BEGIN/COMMIT` within a single `runQuery` call? If not, we may need to use multiple calls and accept partial atomicity. This needs validation during Phase 1.
- **Foreign key ordering in multi-table undo**: When multiple tables are dropped, FK constraints may require specific ordering. The generator should handle this, but complex cross-table dependencies may require schema introspection at undo-generation time.

## Decision Log

| Decision                                             | Reasoning                                                                                                                                                                     |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **v1 = undo-SQL only, defer full dumps**             | Supabase Management API has no native pg_dump endpoint. SQL-based dumps are fragile and slow. Undo-SQL covers the majority of AI-generated schema changes with medium effort. |
| **DDL only, no DML undo**                            | DML undo requires storing old row values — essentially a partial WAL replay system. Fundamentally different (and much harder) problem.                                        |
| **Write-time generation, not rollback-time**         | At write-time we know exactly what SQL was executed. Rollback-time would require schema diffing (essentially pg_diff), which is much harder and less deterministic.           |
| **Independent of migration file setting**            | Undo-SQL is internal Dyad infrastructure, not a developer artifact. Maximum rollback coverage without requiring users to opt into migrations.                                 |
| **Don't store execution results yet**                | YAGNI. DML undo is v2+ and the storage design may change. Keep v1 minimal.                                                                                                    |
| **Per-app rollback strategy (for v2)**               | Different apps have different DB sizes and risk profiles. Prototypes vs. production apps need different strategies.                                                           |
| **Public schema only**                               | Supabase system schemas (auth, storage, vault) have complex interdependencies. Safer to exclude and warn.                                                                     |
| **Conservative generator (null > wrong undo)**       | A missing undo is a UX inconvenience; a wrong undo is data corruption. No safety dump to fall back on in v1.                                                                  |
| **Gate message deletion on full success**            | If DB rollback fails but messages are deleted, user loses conversation context needed for recovery.                                                                           |
| **Reorder: edge functions deploy AFTER DB rollback** | Edge functions may depend on DB schema. Deploying before rollback causes runtime errors during the window between steps.                                                      |
