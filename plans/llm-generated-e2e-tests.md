# LLM-Generated E2E Tests + Tests Panel (Experimental v1)

> Generated by swarm planning session on 2026-06-19

## Summary

We're adding the first slice of a larger auto-testing initiative: the LLM can generate a Playwright end-to-end test for an app it built, and a new **Tests** panel (sibling to Preview / Code / Publish) lets the user run those tests and read the results in-panel. v1 is deliberately lean — **manual, chat-triggered generation only**, **headless only**, and **free for everyone**: generation is exposed both as a free Build-mode `<dyad-generate-test>` tag and as a first-class **`generate_test`** tool in the Pro agent. It ships behind an **experimental** badge and is **instrumented** rather than hard-gated. Its real job is to prove that the LLM can write runnable, mostly-passing Playwright tests against generated apps before we invest in auto-generation.

## Problem Statement

Dyad's core user is a non-expert "vibe coder" building a web app by chatting with an LLM. As the LLM iterates, regressions creep in silently — a new feature breaks an older flow, and the user only discovers it by manually clicking around the preview. They have no cheap way to lock in "this flow works" and catch when a later edit breaks it.

The pain is **real but latent**: app-builder users rarely _ask_ for e2e tests; they ask for working apps. So the primary value of tests here is as a **self-verification safety net for the agent loop** (did my change break something?), and only secondarily as a user-curated artifact. That reframing drives the scope: v1 is a transparency/control surface over a capability the system will increasingly use on its own, and a **reliability experiment** that de-risks the broader auto-testing bet.

## Scope

### In Scope (MVP)

- **Manual `generate_test` capability** — exposed through **both** of Dyad's LLM surfaces so it works for everyone:
  - **Build mode (free, default):** a thin `<dyad-generate-test>` tag — a file-write under the hood — emitted when the user asks in chat ("write an e2e test for the signup flow").
  - **Local Agent mode (Pro):** a first-class **`generate_test`** tool (Zod schema + consent) registered in `TOOL_DEFINITIONS`, which produces the same result via a real tool call.
    Either way the spec is written to the app's `tests/` folder and rendered as a **distinct in-chat card** (flask icon, test title + path) with a **"View in Tests"** deep-link.
- **Tests panel** alongside preview/code/publish, with an **experimental badge**.
- **Test list** of discovered specs, each row with a **hover-revealed Run button** (keyboard-focusable) plus an **always-visible "Run all"** and a **Stop** button while running.
- **Lazy Playwright bootstrap on first run** (not a scaffold change): install `@playwright/test`, install the Chromium browser, write `playwright.config.ts`, append test artifacts to `.gitignore`, add a `test` script — all **streamed + cancellable**, with a clear **offline error**.
- **Headless run** via the existing process-spawn machinery, **gated on the dev server running** (baseURL must resolve).
- **In-panel report:** custom JSON-reporter-driven pass/fail summary per test — status, duration, **error text on failure**, and a **best-effort failure screenshot**.
- **Result-state taxonomy** that separates _infra/inconclusive_ failures (amber) from _assertion_ failures (red), so a flaky test never reads as "you broke your app."
- **Empty / first-run / setup / loading / running / results states** with deliberate copy (the empty state is the most-seen — a CTA, not a blank table).
- **User-initiated "Ask AI to fix this test"** on failures (feeds the failure back into a normal chat turn; **never auto-fixed**).
- **Reliability levers (prompt-only):** instruct the generator to use role/text-based locators + `await expect` auto-waiting, and to add `data-testid` attributes to the components it builds.
- **Instrumentation** of first-run pass-rate and related metrics.

### Out of Scope (Follow-up / Phase 2)

- **Auto-generate-after-each-feature** (the "auto test generation" toggle).
- **Headed mode** and the **"Open browser window"** toggle.
- **Embedded interactive Playwright trace viewer** (`show-trace`) and embedded full HTML report.
- **Generate → run → self-correct-once** reliability loop.
- Multi-browser (Firefox/WebKit), test **editing/rename UI**, **CI/scheduling**, **persisted run history** (DB), visual/snapshot diffs, auth fixtures.

## User Stories

1. As a builder, I ask in chat "write an e2e test for the signup flow" → I see a generation card (pending → done) → the test appears in the Tests tab.
2. As a builder opening the Tests tab for the first time with no tests, I see a clear empty state explaining what tests do and a CTA — not a blank table.
3. As a builder, I hover a test and click Run (or click the always-visible "Run all") → it runs headless → I watch live progress ("2/5 passed") and can Stop.
4. As a builder, after a run I see a pass/fail summary per test with error text on failures, and each failure is clearly marked **infra/amber** (setup broke) vs **assertion/red** (the app behavior failed).
5. As a builder whose test went red, I click **"Ask AI to fix this test"** → the failure is sent back into chat for a user-initiated fix (never auto-fixed).

> Implicit requirement (not a story): first run with no Playwright installed triggers a streamed, cancellable one-time setup state — never a cryptic hang or error.

## UX Design

### User Flow

1. **Generate:** user asks in chat → LLM emits `<dyad-generate-test>` (Build mode) or calls the `generate_test` tool (Pro agent) → a distinct test card renders (pending → done) → the spec is written to the app's `tests/` folder and appears in the Tests tab. The in-chat card is the **primary discovery path** to the panel (the panel never auto-focuses).
2. **Discover:** the user opens the new **Tests** tab (an overflow tab, with an experimental pill).
3. **Run:** hover a row → Run; or "Run all". First run triggers the streamed Playwright setup.
4. **Watch:** live streaming progress — per-row spinner + header counter "3 / 12 passed · 1 failed" + Stop.
5. **Review:** per-test pass/fail summary; expand a failure for error text + screenshot + "Ask AI to fix this test".

### Key States (with copy)

- **Empty (no specs):** title "No tests yet"; body "Ask the AI in chat to write a test for a feature — generated tests show up here. They're a starting point you can review and re-run." (No empty table.)
- **First-run (specs exist, never run):** rows show gray "Not run yet"; "Run all" is the primary CTA; no banner.
- **Setup / first-run Playwright install (lazy, on first Run):** title "Setting up testing…"; streamed sub-line of the real install / `playwright install chromium` output (reuse the Console streaming model); a **Cancel** button; offline failure: "Couldn't download the test browser. Check your connection and try again." + **Retry**. **v1-critical** — never a silent hang.
- **Dev-server-not-running (Run pressed while app stopped):** gate the run; inline "Start the app to run tests" + a **Start** button. Don't spawn a test against a dead baseURL.
- **Running:** per-row spinner on the active test + header counter (aria-live polite) + **Stop**; a collapsible raw-output drawer for power users. Not one big blocking spinner.
- **Results:** summary list per the state model below; failure rows expand to error text + screenshot + "Ask AI to fix this test".
- **Loading (panel mount / discovering specs):** reuse the existing PublishPanel spinner + label.

### Result-State Model

Every state carries a **text/aria label** (never color-only):

| State                | Color       | Label / copy                                                                                                      |
| -------------------- | ----------- | ----------------------------------------------------------------------------------------------------------------- |
| Passed               | green check | "Passed"                                                                                                          |
| Failed (assertion)   | red         | "Test failed — your app may not match the test"                                                                   |
| Inconclusive (infra) | amber       | "Couldn't run — needs a fix to the test" (Playwright not installed, dev server down, selector-not-found, timeout) |
| Not run              | gray dot    | "Not run yet"                                                                                                     |
| Running              | spinner     | "Running"                                                                                                         |

The amber-vs-red wording split is the core of "informative, not broken": amber points the user at the **test**, red at the **app**.

### Interaction Details

- **Hover Run button:** reuse the existing pattern — parent row `group`, button `opacity-0 group-hover:opacity-100 focus-visible:opacity-100`. Because it's the only per-test entry point, the `focus-visible` and a **real `<button aria-label="Run test: …">`** are non-negotiable; **"Run all" is always visible** so nothing is unreachable without hover.
- **No mode/settings controls** in the header (no headed/headless toggle in v1) — just title + experimental pill + "Run all" + Stop (Stop only while running).
- **No auto-focus / auto-open** of the panel on any result — an unbidden red panel reads as "Dyad is broken."
- **"AI-generated tests are a starting point"** framing in the empty/first-run states sets the expectation that red may mean the test, not the app.

### Accessibility

- Real focusable `<button>`s with `aria-label`s for Run / Run all / Stop / "Ask AI to fix this test".
- Status conveyed by text + icon, not color alone (colorblind/contrast safe).
- Live progress via `aria-live="polite"` on the counter.
- Touch/no-hover safety net: "Run all" always visible; per-row Run reachable via keyboard focus.
- Long test titles/paths truncate with a tooltip rather than wrapping the row.

### Consistency Notes

Almost nothing here needs net-new visual language:

- In-chat generated-test card → the existing DyadWrite/DyadEdit "pending → done" card pattern.
- Live run output → the existing Console `app:output`/`app:output-batch` streaming model.
- Hover Run → the existing ChatTabs hover + focus-visible pattern.
- Empty/loading → existing PublishPanel patterns.
- Experimental badge → the existing yellow pill (`TemplateCard.tsx`) / `Badge variant="secondary"` (`SecurityPanel.tsx`).

## Technical Design

### Architecture

A test is just a file, so **generation is prompt + file-write**, not a new execution primitive. Generation is exposed via two thin entry points that converge on the same file-write: the free Build-mode `<dyad-generate-test>` tag and the Pro Local-Agent `generate_test` tool. Everything downstream — the panel, running tests, the report — is **free-tier**; the only `src/pro/` touch in v1 is the one `generate_test` tool file.

`<dyad-generate-test>` tag / `generate_test` tool (chat) → file written to the app's `tests/` → discovered + listed in `TestsPanel` → `runAppTests` IPC handler spawns `npx playwright test --reporter=json` against the running dev server → JSON parsed into ephemeral jotai state → rendered as the in-panel report.

### Components Affected

- `src/atoms/appAtoms.ts` — add `"tests"` to the `PreviewMode` union (drives `previewModeAtom`).
- `src/components/preview_panel/PreviewToolbar.tsx` — add `tests` to `OVERFLOW_MODES` + a `modeMeta` entry (icon/label/testId) + the experimental badge.
- `src/components/preview_panel/PreviewPanel.tsx` — add the `previewMode === "tests" ? <TestsPanel/> : …` branch in the content router.
- **New** `src/components/preview_panel/TestsPanel.tsx` — list, hover-Run, Run-all, Stop, streamed running state, results with the infra-vs-assertion taxonomy, failure screenshot, collapsible raw output. (Main UI work; reference `PlanPanel.tsx` / `Problems.tsx` / `PublishPanel.tsx`.)
- `src/atoms/previewRuntimeAtoms.ts` — per-app test atoms (Maps keyed by `appId`, mirroring `previewRunStateByAppIdAtom`).
- `src/prompts/system_prompt.ts` — test-writing guidance: write specs to `tests/`, use role/text locators + `await expect` auto-waiting, emit `data-testid`s on components built. (Reused as the `generate_test` tool's description.)
- `src/ipc/utils/dyad_tag_parser.ts` — add `getDyadGenerateTestTags()` (clone of `getDyadWriteTags()`).
- `src/ipc/processors/response_processor.ts` — route `<dyad-generate-test>` through the existing file-write path in `processFullResponseActions()`.
- **New** `src/components/chat/DyadGenerateTest.tsx` (fork of `DyadWrite.tsx`) + one `case "dyad-generate-test"` in `src/components/chat/DyadMarkdownParser.tsx` → the distinct card + "View in Tests" deep-link.
- **New** `src/pro/main/ipc/handlers/local_agent/tools/generate_test.ts` — the first-class `generate_test` Local Agent tool (`name`/`description`/`inputSchema`/`execute`/`buildXml`, modeled on `run_type_checks.ts` / `generate_image.ts`); writes the spec file and emits the same `<dyad-generate-test>` card. Registered in `src/pro/main/ipc/handlers/local_agent/tool_definitions.ts` (`TOOL_DEFINITIONS`). **The one `src/pro/` file v1 adds.**
- **New** `runAppTests` IPC handler + contract + typed client (mirror `capacitor_handlers.ts` + `src/ipc/utils/simpleSpawn.ts`).
- **New** bootstrap helper util (detect/install Playwright, write config, patch `.gitignore` + `package.json`).
- `scaffold/.gitignore` — see "must-not-forget" below.

### Data Model Changes

**None persisted.** Run results live in **ephemeral jotai state keyed by `appId`** (re-runnable anytime). Test files on disk in the app repo are the durable, versioned artifact. **No DB table, no Drizzle migration.** (Persisted run history is a later "nice to have.")

### API Changes

- `runAppTests({ appId: number; testFile?: string })` → spawns `npx playwright test [testFile] --reporter=json` with `cwd: getDyadAppPath(app.path)`; streams progress over the existing `app:output`-style channel; returns parsed JSON results. No `headed`/mode field in v1.
- Optional companion `listAppTests({ appId })` → glob `tests/*.spec.ts` to populate the list.
- No engine/external API changes.

### Generated-app Playwright config (templated by bootstrap)

`playwright.config.ts` at the app root: `baseURL` = the **already-running dev-server proxy URL**, `workers: 1`, `reporter: 'json'`, `screenshot: 'only-on-failure'`. (Trace retention + the `html` reporter come with the Phase-2 trace viewer.) **Do not** use Playwright's `webServer` — point baseURL at the running proxy to avoid a double-start.

### baseURL source

The running dev server's **proxy URL**, surfaced via `[dyad-proxy-server]started=[URL]` (`app_runtime_service.ts`) → `appUrlByAppIdAtom` (`previewRuntimeAtoms.ts`). Computed deterministically from `getAppPort(appId) = 32100 + appId % 10000` / `getAppProxyPort`. The dev server **must be running** — gate the Run button (or auto-start via `executeApp` first, à la the capacitor handler's build-before-sync).

### Must-not-forget engineering items

1. **`.gitignore` fix (required):** the scaffold `.gitignore` has `node_modules`/`dist` but **not** test artifacts. Bootstrap must append `/test-results/`, `/playwright-report/`, `/playwright/.cache/`, `.last-run.json`, or multi-MB reports/traces pollute the user's git history.
2. **Error taxonomy in `runAppTests`:** distinguish **infra failure** (Playwright missing, browser-download failed, dev server down, baseURL unreachable, spawn error) → **amber** from **assertion failure** (JSON `status: "failed"`) → **red**. Core, not polish.
3. **Dev-server-running gate** before any run.
4. **baseURL = the running proxy URL** (not Playwright `webServer`).
5. **Ephemeral jotai state keyed by `appId`** — no DB.
6. **Generation and execution are separate explicit actions** — never auto-run freshly generated code (security posture). "Ask AI to fix this test" is user-initiated only.
7. **Stream first-run bootstrap output** through the existing channel into the panel's collapsible raw-output, with **Stop/cancel**.
8. **Instrumentation hook** at the JSON-parse point to capture first-run pass/fail counts for the pass-rate metric.

| **Selector stability / silent app mutation** — generating a test may require adding `data-testid`s, so "write a test" can edit app code | M          | M      | Known v1 limitation; surfaced via the file-write cards; likely driver of flake/amber — revisit if flake is high                                                                                |
| **First-run install latency / offline** — Chromium download is hundreds of MB, needs network                                            | M          | M      | Designed, streamed, cancellable setup state; explicit offline error + Retry                                                                                                                    |
| **Security** — running arbitrary LLM-generated test code + a real browser locally                                                       | M          | M-H    | Generate and run are separate explicit user actions (never auto-run); keep tests in a known dir; route install/run through the existing firewall posture; experimental disclaimer              |
| **Repo clutter** — `@playwright/test` + config land in the user's app/package.json                                                      | L          | L      | Accepted (lazy, only on use); transparency in the chat card; gitignore reports                                                                                                                 |
| **Dev-server coupling** — tests need the app running for baseURL                                                                        | L          | L      | Gate the Run button; offer Start                                                                                                                                                               |

## Success Metrics (instrumented, not hard-gated)

1. **First-run pass-rate** _(primary, make-or-break)_ — of generated tests, % that run and pass on first execution without edits.
2. **Flake rate** — % of results that flip across re-runs with no code change (trust-killer if high).
3. **Activation** — % of active apps that generate ≥1 test (is there demand at all?).
4. **Re-run rate** — % of tests run ≥2× (signals the regression-safety-net value is landing).

- **Guardrail:** no increase in preview/app-run crash rate or chat-turn latency; watch **infra-failure (amber) rate** separately — a high amber rate means our bootstrap/runner is broken, not the LLM.

## Open Questions

These do not block v1; resolve during implementation or Phase 2:

- **Auto-gen tiering:** is auto-generate-after-each-feature free or **Pro**? (Deferred with auto-gen; the natural monetization hook for the autonomous behavior.)
- **Selector stability:** if flake is driven by brittle locators, how aggressively should the generator add `data-testid`s to app code, and is that mutation acceptable as a side effect of "write a test"?
- **Pull-back threshold:** the exact first-run pass-rate / flake-rate line that triggers iterate-or-kill after launch.
- **Spec breadth:** one happy-path test per feature vs broader coverage (affects whether the list needs grouping/virtualization sooner).

## Decision Log

- **Manual-first, auto-gen deferred** _(human decision)_ — ship chat-triggered generation in v1; auto-after-each-feature is the highest-cost/highest-risk piece (extra LLM turn per feature, noise, flaky-failure thrash) and shouldn't gate the experiment. All three roles agreed.
- **Headless only, no headed/toggle/window in v1** _(human decision)_ — the spec's "headed mode inside the preview iframe" is **technically impossible**: Playwright's headed browser is its own OS window driven by the automation engine and cannot render into the existing app iframe. Rather than ship a confusing separate-window toggle, v1 is headless-only. The honest in-panel "watch it run" (trace viewer) and a real headed window are Phase 2.
- **`generate_test` exposed as BOTH a free Build-mode tag and a first-class Pro tool, both in Phase 1** _(updated 2026-06-19 per user request; originally "free & lightweight tag only")_ — a test is just a file, so the lightweight free path is a thin `<dyad-generate-test>` Build-mode tag (file-write under the hood; ~1 parser fn + 1 switch case + 1 card) whose job vs a plain write is the **distinct chat card + "View in Tests" deep-link**. On top of that, v1 now also adds the literal **`generate_test`** tool in the Pro Local Agent system (Zod schema + `execute`), so in agent mode the LLM truly _calls_ `generate_test`. Both converge on the same file-write + card, so the panel/run/report stay shared and free. **Why keep both:** Build mode cannot make tool calls, so the free tag is required to keep manual generation free for everyone, while the Pro tool satisfies the literal "the LLM calls the generate_test tool" intent. **Trade-off accepted:** v1 now touches one `src/pro/` file (previously zero) and sizing nudges up slightly (still ~medium). A bare `generate_test` (no `dyad-` prefix) Build-mode tag was rejected to keep the existing `<dyad-*>` tag convention; the tag is `<dyad-generate-test>` and the Pro tool is `generate_test`.
- **Ship & instrument, not hard-gate** _(human decision)_ — release behind the experimental badge; instrument first-run pass-rate as the primary metric; commit to a post-launch pull-back trigger. "Experimental" is not a license to ship something that mostly fails, so keep an informal dev smoke-test bar.
- **Trace viewer → Phase 2** _(team tie-break)_ — UX and PM both placed the embedded trace viewer in Phase 2 to honor the "headless only now" minimalism; v1's "see what happened" is the **error text + best-effort failure screenshot**. Eng confirmed the trace viewer (`show-trace`) is cheap to pull forward later, so it's a low-cost Phase-2 add (and the designated first thing to pull in if we want more).
- **Report = JSON reporter → custom panel UI** _(team)_ — drives the list + per-test status directly and matches existing panels; avoids serving Playwright's HTML bundle (an extra moving part) in v1.
- **Tests are real files committed to the user's app** _(team)_ — consistent with Dyad's "the user owns their code" model, versioned, portable to the user's own CI; corralled in one `tests/` folder and managed from the Tests panel. Reports/artifacts are gitignored.
- **Ephemeral jotai run state, no DB** _(team)_ — results are transient and re-runnable; the spec files are the durable artifact. Avoids a Drizzle migration.
- **Reliability levers 1 + 2 in v1; lever 3 later** _(team)_ — prompt for role/text locators + auto-waiting (biggest, trivial lever) and `data-testid` emission (small); hold the generate-run-self-correct loop for Phase 2, gated on pass-rate data (also avoids auto-fix thrash).
- **Bootstrap transparency in the chat card** _(team)_ — disclose the added `@playwright/test` + config in the generation/setup card, and give the Chromium download its own streamed progress line; no modal, no settings gate.
